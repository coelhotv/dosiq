## Entregas

### 1. Slim select para DashboardProvider + HealthHistory calendar
- `getByDateRangeSlim`: select mínimo sem relações (~60 bytes/log vs ~315)
- `getByMonthSlim`: só `medicine.name` para calendário (~80 bytes/log)
- `cachedServices.js`: wrappers SWR para os novos métodos

### 2. Zod validation em todos os 8 métodos de leitura do logService
- Schemas reutilizáveis: `dateRangeSchema`, `monthSchema`, `limitSchema`, `offsetSchema`
- `safeParse()` com fallback gracioso (retorna `{ data: [], total: 0 }`)

### 3. parseLocalDate() em getByDateRange/Slim + getByMonth/Slim
- Substituiu templates UTC hardcoded (`` `${date}T00:00:00.000Z` ``)
- Corrige off-by-one day em GMT-3 (R-020, R-131)
- Aplicado nos 4 métodos que fazem queries de range de data

### 4. getUserId() cache com promise coalescence
- Cache em memória: `_cachedUserId` + `_userIdPromise`
- Coalescência: múltiplas chamadas simultâneas compartilham mesma promise
- Auto-invalidação via `onAuthStateChange` (SIGNED_IN/SIGNED_OUT)
- **Impacto:** 13 → 1 auth roundtrip (~8s economia em 4G)

### 5. calculateStreaks optimization
- String comparison em vez de `new Date()` para datas no hot loop
- Pre-parse de protocolos: `protocols.map(p => ({ startStr, endStr, timesPerDay }))`
- Guard: `MAX_ITERATIONS = 91` no loop de streak (antes era `while(true)`)
- **Impacto:** CPU `parseLocalDate` de 71.3% → negligível

### 6. Fix coluna `status` inexistente
- `getByDateRangeSlim` incluía `status` no select — coluna não existe em `medicine_logs`
- Causava HTTP 400 + widget "últimas doses" vazio no Dashboard

### 7. Execution spec para Dashboard first load
- `plans/EXEC_SPEC_DASHBOARD_FIRST_LOAD.md` — 6 sprints (D1-D6)
- Target: 25 → ≤12 GET queries, ~15s → <5s em 4G
- Ordem: D5 → D1 → D2 → D3 → D4 → D6

## Lições Aprendidas

- **Auth roundtrip é o gargalo silencioso:** `supabase.auth.getUser()` SEMPRE faz HTTP (sem cache client). Em app com 10+ services, cada `getUserId()` = 1 roundtrip. Promise coalescence é obrigatória.
- **String comparison >> Date objects em hot loops:** Chrome trace revelou que 71.3% do CPU time era `parseLocalDate()` em `calculateStreaks`. YYYY-MM-DD é lexicographically sortable — não precisa de parsing.
- **UTC hardcoded em queries Supabase é bug latente:** `` `${date}T00:00:00.000Z` `` funciona em UTC mas falha em GMT-3. `parseLocalDate().toISOString()` é o único padrão correto.
- **Schema drift causa erros silenciosos:** `status` nunca existiu em `medicine_logs` mas estava no select. HTTP 400 sem mensagem clara na UI.
- **Zod validation em leitura, não só escrita:** Parâmetros inválidos em GET methods causam queries ineficientes ou erros no Supabase.

## Rules/Anti-Patterns Registrados

- R-128: Promise coalescence para auth roundtrips
- R-129: String comparison para datas em hot loops
- R-130: Zod validation em TODOS os métodos de leitura
- R-131: parseLocalDate em TODAS as queries de data ao Supabase
- AP-P14: getUserId sem cache (13 roundtrips)
- AP-P15: new Date() em hot loop (71.3% CPU)
- AP-P16: Template UTC hardcoded em queries
- AP-P17: select de coluna inexistente
