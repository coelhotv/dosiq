## AP-096: CSS Deletion Without Verifying className Usage in JSX

### O Problema

Durante limpeza de CSS (S17.9), um arquivo `LandingPrototype.css` foi deletado após verificação grep que mostrou "zero imports":

```bash
grep -r "LandingPrototype" src/ --include="*.jsx"
# Resultado: nenhum match
# Conclusão: arquivo orphaned → delete
```

**MAS:** Landing.jsx usa 100+ CSS classes com prefixo `lp-*` que eram definidas nesse arquivo:
- `.lp-page`, `.lp-header`, `.lp-shell`, `.lp-brand`, `.lp-nav`, `.lp-hero`, `.lp-pill`, `.lp-btn`
- `.lp-alert-card`, `.lp-progress`, `.lp-dose-list`, `.lp-badge`, `.lp-emergency-card`
- `.lp-floating-bot`, `.lp-section`, `.lp-card-grid`, `.lp-footer`
- E 80+ mais...

O arquivo CSS era referenciado via `className` attribute, não via import statement.

### Por Que Aconteceu

1. **Grep incompleto:** Apenas procura por `"LandingPrototype"` (string literal no import)
2. **Não procura className:** Não escaneia por `className="lp-..."` em JSX
3. **Assumiu import obrigatório:** CSS deve sempre ser importado explicitamente (falso para className-driven CSS)

### Impacto

- **Regressão total:** Landing page layout 100% quebrado (todas as classes não encontradas)
- **Silent failure:** Erro só visível em runtime/visual test, não em lint/build
- **Recovery cost:** git restore + novo commit + re-review + re-merge

**Severidade:** CRITICAL (layout regression em marketing page — abandono de usuários)

### Prevenção

**Antes de deletar qualquer arquivo CSS:**

```bash
# 1. Verificar imports
grep -r "LandingPrototype\|LandingPrototype.css" src/ --include="*.jsx"

# 2. Verificar className usage (OBRIGATORIO)
grep -r "className=\".*[a-z-]*\"\|className='[a-z-]*'" src/ --include="*.jsx" | \
  grep -E "lp-|landing-|prototype-"  # ou prefixo do arquivo em questão

# 3. Verificar em template strings
grep -r "className=.*\`\|className=.*{" src/ --include="*.jsx" | head -20

# 4. Se qualquer match encontrado: arquivo NAO eh orphaned
```

### Padrão Positivo Relacionado

**R-086 (Grep Before Delete)** — Necessário, MAS incompleto. Precisa:
1. Grep imports: `grep -r "from.*file\|import.*file"`
2. Grep className: `grep -r "className=.*prefix-"`
3. Grep CSS modules: `grep -r "file\.className"`
4. Manual visual scan de landing pages / marketing / high-visibility areas

### Exemplo Correto

```bash
#!/bin/bash
# Antes de deletar LandingPrototype.css

FILE="LandingPrototype.css"
PREFIX="lp-"  # inferido do nome e inspeção do arquivo

# Check 1: Is it imported?
grep -r "import.*$FILE\|from.*$FILE" src/ --include="*.jsx" && \
  echo "ERROR: Still imported" && exit 1

# Check 2: Are its classes used?
grep -r "className=.*$PREFIX\|className='.*$PREFIX" src/ --include="*.jsx" && \
  echo "ERROR: Classes still used" && exit 1

# Check 3: Manual inspection of output
grep -r "$PREFIX" src/ --include="*.jsx" | head -20

# Only then:
rm "src/shared/styles/$FILE"
```

### Regra de Ouro

**Nunca confie em grep sozinho para verificação de orfandade CSS.**
- CSS pode ser referenciado via: imports, className (string), className (template literal), CSS modules, Tailwind-style utilities
- Sempre fazer verificação visual de high-visibility components (Landing, Marketing, Auth flows)
- Considerar ferramentas de análise estática (AST parsing) para production-critical deletions

### Lição Aprendida (S17.9)

Landing.jsx não tinha import explícito — CSS era carregado via `className` attributes.
Nomenclatura "Prototype" deveria ser refatorada em sprint futuro (nota: essas classes vieram de A/B test vencedor — apenas renomear, não deletar).
