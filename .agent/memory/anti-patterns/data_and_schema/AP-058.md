## Problema

Em **Zod v4**, `z.record(valueType)` com argumento único define esse argumento como `keyType`, deixando `valueType = undefined`. O schema **parece** funcionar (construção não lança erro, e objetos vazios passam), mas explode com `Cannot read properties of undefined (reading '_zod')` ao tentar parsear qualquer objeto com chaves reais.

## Exemplo quebrado

```js
// ❌ Zod v4 — valueType=undefined → TypeError em runtime com dados reais
provider_metadata: z.record(z.any()).default({})
// Funciona para {} mas explode para { tickets: [...] }
```

## Fix

```js
// ✅ Sempre dois argumentos em Zod v4
provider_metadata: z.record(z.string(), z.unknown()).default({})
```

## Impacto real

Sprint 8.2/8.3: todo disparo de push mobile falhava ao persistir `notification_log` no banco porque `provider_metadata` sempre continha `{ tickets: [...] }`. Notificações chegavam no dispositivo mas nunca apareciam na Central de Avisos.

## Prevenção

- Sempre fornecer dois argumentos: `z.record(keySchema, valueSchema)`
- Preferir `z.unknown()` a `z.any()` para valores de estrutura incerta
- Testar schemas com objetos **não-vazios** — o bug não dispara com `{}`
